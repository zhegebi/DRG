# DRG 入组智能体系统 —— 认证逻辑

## 一、架构总览

本系统采用 **JWT（JSON Web Token）** + **HttpOnly Refresh Token** 的双 Token 认证方案。

```
┌─────────────────┐         ┌──────────────────────────────┐
│  前端 (Vue 3)   │         │  后端 (FastAPI / Python)      │
│                 │  HTTP   │                              │
│  AuthPage.vue   │ ──────→ │  /api/auth/signup   (注册)    │
│  auth.ts (Pinia)│ ←────── │  /api/auth/login    (登录)    │
│  router/index.ts│         │  /api/auth/refresh  (刷新)    │
│                 │         │  /api/auth/logout   (登出)    │
└─────────────────┘         └──────────────────────────────┘
                                      │
                                      ▼
                            ┌─────────────────┐
                            │   SQLite 数据库   │
                            │   User 表        │
                            └─────────────────┘
```

## 二、后端认证逻辑

### 2.1 密码加密

- **算法**：Argon2（`passlib` + `argon2-cffi`）
- **配置参数**：time_cost=3, memory_cost=65536, parallelism=4
- 密码从不以明文存储，数据库中仅存 `password_hash`

### 2.2 Token 机制

| Token 类型 | 存储位置 | 有效期 | 内容 |
|:---|:---|:---:|:---|
| Access Token | 前端 localStorage | 30 分钟 | `{ sub: 用户ID, exp: 过期时间 }` |
| Refresh Token | 后端 HttpOnly Cookie | 7 天 | `{ sub: 用户ID, type: "refresh", exp: 过期时间 }` |

**签名算法**：HS256  
**密钥**：环境变量 `SECRET_KEY`，默认 `your-secret-key-change-in-production`

### 2.3 API 端点

#### POST /api/auth/signup — 注册

```
请求体: { username: str, email: str, password: str }
响应:   { access_token: str, token_type: "bearer", username: str, email: str }
```

流程：
1. 检查 username / email 是否已存在 → 400
2. 校验 username 和 password 格式 → 422
3. 加密密码，创建 User 记录写入数据库
4. 生成 Access Token + Refresh Token
5. Refresh Token 写入 HttpOnly Cookie
6. 返回 Access Token（含用户名和邮箱）

#### POST /api/auth/login — 登录

```
请求体: { type: "email"|"username", identifier: str, password: str }
响应:   { access_token: str, token_type: "bearer", username: str, email: str }
```

流程：
1. 根据 `type` 按邮箱或用户名查找用户
2. 校验密码（Argon2 verify）→ 401
3. 生成 Access Token + Refresh Token
4. Refresh Token 写入 HttpOnly Cookie
5. 返回 Access Token（含用户名和邮箱）

#### POST /api/auth/refresh — 刷新 Access Token

```
请求: 从 Cookie 读取 refresh_token
响应: { access_token: str, token_type: "bearer" }
```

流程：
1. 从请求 Cookie 中读取 refresh_token
2. 验证 Token 签名和 `type: "refresh"` 字段 → 401
3. 检查对应用户是否仍存在
4. 签发新的 Access Token

#### POST /api/auth/logout — 登出

- 清除 HttpOnly Cookie 中的 Refresh Token

### 2.4 请求认证守卫

后端通过 FastAPI 的 `Depends` 机制保护需要登录的接口：

- **`get_current_user`**（强制认证）：从请求头提取 Bearer Token，解码 JWT 获取用户 ID，从数据库查询并返回 User 对象。Token 无效或用户不存在时返回 401。
- **`get_optional_user`**（可选认证）：Token 存在时同 `get_current_user`，不存在时返回 None 不报错。

使用方式：
```python
@router.get("/protected")
async def protected_route(user: User = Depends(get_current_user)):
    ...
```

### 2.5 用户表结构

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| id | Integer, Primary Key | 用户唯一标识 |
| username | String | 用户名 |
| email | String | 邮箱 |
| password_hash | String | Argon2 加密后的密码哈希 |

## 三、前端认证逻辑

### 3.1 Auth Store（Pinia）

文件：`client/src/stores/auth.ts`

**状态：**
| 字段 | 类型 | 说明 |
|:---|:---|:---|
| `accessToken` | `ref<string \| null>` | 当前 Access Token，初始化时从 localStorage 恢复 |
| `user` | `ref<User \| null>` | 当前用户信息，初始化时从 localStorage 恢复 |
| `isLoading` | `ref<boolean>` | 登录/注册请求中 |
| `error` | `ref<string \| null>` | 错误信息 |
| `isAuthenticated` | `computed<boolean>` | 是否有 Access Token |

**关键方法：**

**`setAuth(token, userData?)`**
- 保存 token 到 state 和 localStorage（`access_token`）
- 保存用户信息到 state 和 localStorage（`user_info`，JSON 格式）
- 若无 userData，则从 JWT 解码出用户 ID 作为降级

**`clearAuth()`**
- 清除 state 中的 token 和用户信息
- 清除 localStorage 中的 `access_token` 和 `user_info`

**`login(identifier, password, type)`**
- 调用 `/api/auth/login`
- 成功 → `setAuth(token, { username, email })`
- 401 → 设置错误信息"用户名或密码错误"

**`signup(username, email, password)`**
- 调用 `/api/auth/signup`
- 成功 → `setAuth(token, { username, email })`
- 400 → "用户名或邮箱已被注册"
- 422 → "用户名或密码格式无效"

**`logout()`**
- 调用 `/api/auth/logout`
- 无论成功失败 → `clearAuth()`

**`refreshToken()`**
- 调用 `/api/auth/refresh`
- 成功 → 更新 localStorage 中的 `access_token`
- 失败 → `clearAuth()`

### 3.2 页面刷新恢复流程

```
页面加载
  │
  ├─→ localStorage.getItem("access_token") → 恢复 token
  │
  ├─→ localStorage.getItem("user_info") → 恢复用户信息
  │     │
  │     └─→ 不存在但有 token → 从 JWT 解码 user.id
  │
  └─→ isAuthenticated = (token 存在)
```

### 3.3 前端路由守卫

文件：`client/src/router/index.ts`

```
路由访问
  │
  ├─→ requiresAuth = true 且 未登录 → 重定向到 /auth?redirect=原路径
  │
  ├─→ requiresGuest = true 且 已登录 → 重定向到 /
  │
  └─→ 否则正常放行
```

路由元信息：

| 路由 | requiresAuth | requiresGuest |
|:---|:---:|:---:|
| `/`（首页） | false | - |
| `/auth`（登录页） | - | true |
| `/DRG`（DRG入组） | true | - |
| `/DocGen`（文档生成） | true | - |

### 3.4 登录页面

文件：`client/src/views/Auth/AuthPage.vue`

- **双模式**：登录 / 注册，通过 Tab 切换
- **登录方式**：邮箱登录 / 用户名登录
- **前端校验**：邮箱格式、用户名长度≥3、密码长度≥6、确认密码一致性
- **成功后**：检查 URL 中 `redirect` 参数跳回原页面，无则跳首页

### 3.5 HTTP 客户端

Access Token 自动附加到每个 API 请求中：

```typescript
// main.ts
client.setConfig({
  auth: () => localStorage.getItem('access_token') ?? undefined,
})
```

## 四、完整认证流程

### 4.1 注册流程

```
用户填写注册表单
  │
  ├─→ 前端校验（用户名≥3字符、邮箱格式、密码≥6位、两次密码一致）
  │
  ├─→ POST /api/auth/signup
  │     │
  │     ├─→ 后端校验用户名/邮箱唯一性
  │     ├─→ Argon2 加密密码
  │     ├─→ 写入数据库
  │     ├─→ 生成 Access Token + Refresh Token
  │     ├─→ 设置 Refresh Token Cookie
  │     └─→ 返回 { access_token, username, email }
  │
  ├─→ 前端 setAuth()
  │     ├─→ token 存入 state + localStorage
  │     └─→ user 存入 state + localStorage
  │
  └─→ 跳转至首页（或 redirect 参数指定的页面）
```

### 4.2 登录流程

```
用户填写登录表单
  │
  ├─→ 前端校验（邮箱/用户名非空、密码非空）
  │
  ├─→ POST /api/auth/login
  │     │
  │     ├─→ 后端查找用户
  │     ├─→ Argon2 校验密码
  │     ├─→ 生成 Access Token + Refresh Token
  │     ├─→ 设置 Refresh Token Cookie
  │     └─→ 返回 { access_token, username, email }
  │
  ├─→ 前端 setAuth()
  │
  └─→ 跳转至首页或原页面
```

### 4.3 页面刷新恢复流程

```
浏览器刷新
  │
  ├─→ Pinia Store 初始化
  │     ├─→ localStorage.getItem("access_token") → accessToken
  │     └─→ localStorage.getItem("user_info") → user
  │
  ├─→ 路由守卫检查
  │     └─→ 当前路由需要认证且 token 存在 → 放行
  │
  └─→ 页面正常渲染，导航栏显示用户头像
```

### 4.4 登出流程

```
用户点击退出登录
  │
  ├─→ POST /api/auth/logout（清除 Refresh Token Cookie）
  │
  ├─→ clearAuth()
  │     ├─→ 清除内存中 token 和 user
  │     └─→ 清除 localStorage 中 access_token 和 user_info
  │
  └─→ 若当前页面需要认证 → 跳转首页
```

## 五、安全配置

| 配置项 | 值 | 说明 |
|:---|:---:|:---|
| `AUTH_SECRET_KEY` | 环境变量 `SECRET_KEY` | JWT 签名密钥，生产环境必须更换 |
| `AUTH_ALGORITHM` | HS256 | JWT 签名算法 |
| `AUTH_ACCESS_TOKEN_EXPIRE_MINUTES` | 30 | Access Token 有效期 |
| `AUTH_REFRESH_TOKEN_EXPIRE_DAYS` | 7 | Refresh Token 有效期 |
| 密码加密 | Argon2 | 抗 GPU 暴力破解 |
| Refresh Token | HttpOnly Cookie | 不可被 JavaScript 读取，防 XSS |
| 密码最小长度 | 6 | 前端+后端双重校验 |
| 密码最大长度 | 20 | |
